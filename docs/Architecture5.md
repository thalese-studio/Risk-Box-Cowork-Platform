# Risk Box：金融通訊協議層 AI 風險感測節點 (Risk-Sensing Node)

## 版本資訊
- **版本**：v2.0.0 (Compliance Stable)
- **技術層級**：OSI L2 / L3 協議層
- **監測模式**：Pure Observer (無 IP 隱身監聽)
- **核心目標**：即時識別 ISO-8583 / ISO-20022 異常交易特徵，提供風險警示，不介入銀行交易決策。

---

## 1. 系統定位與核心原則

Risk Box 是一款部署於銀行資料中心網路骨幹的風險感測組件，專為高合規要求環境設計。

* **L2/L3 透明接入**：透過 Port Mirroring (SPAN) 或 TAP 接入，對現有網路架構完全透明，無須修改路由或配置代理。
* **非侵入式監測**：系統定位為「旁路監聽者」，僅提供風險評分與警示（Alert），不具備阻斷功能，確保銀行業務連續性。
* **個資零碰觸**：嚴格針對 ISO 標準報文欄位進行特徵提取，不存儲任何 PII (個人可識別資訊)，所有敏感欄位在內存中即完成 Hash 化。
* **數據主權歸屬**：所有黑/白名單及保護名單由銀行端預載，Risk Box 僅作為執行與比對引擎，不自行定義名單。

---

## 2. 技術架構總覽

### 2.1 數據流轉邏輯
1. **流量擷取層**：從交換機獲取 L2/L3 原始封包。
2. **協議還原引擎**：將碎片封包還原成完整的 TCP/IP 流並重組為金融報文。
3. **特徵矩陣化**：提取 ISO 協議特定位元組（Bitmaps/Data Elements），轉換為機器學習可讀的特徵向量。
4. **AI 風險演算**：結合預載名單與動態行為模型產出 $R\_score$。
5. **警示推送**：透過獨立的 Out-of-band 網路將結果推送到銀行風控管理後台。

### 2.2 部署模式
- **隱身模式 (No-IP)**：監測網口不配置 IP 地址，在網路層對攻擊者不可見。
- **高可用性 (HA)**：支援雙機 Heartbeat 監控，確保監測不中斷。

## 3. 協議解析引擎 (ISO Standard Mapping)

針對金融通訊協議進行深度掃描，僅提取非感性交易維度：

| 監控維度 | ISO-8583 (ATM/卡) | ISO-20022 (匯款) | 異常行為識別重點 |
| :--- | :--- | :--- | :--- |
| **交易金額** | DE-4 | IntrBkSttlmAmt | 偵測規避限額的小額拆分交易（Structuring） |
| **時間戳記** | DE-7 | CreDtTm | 識別非營業時間或極短間隔的高頻交易 |
| **商戶類型** | DE-18 | Purp / Prtry | 監控資金是否流向異常分類之商戶 |
| **收發機構** | DE-32 / DE-33 | InstgAgt / InstdAgt | 偵測異常的跨行清算路徑與非法跳板 |
| **對手帳戶** | DE-102 (Hash) | CdtrAcct (Hash) | 比對黑名單或受保護名單(如高齡、弱勢帳戶) |
| **終端識別** | DE-41 | PmtTpInf | 偵測同一實體設備下的多帳戶異動行為 |

## 4. AI 自主成長與行為建模

Risk Box 內建自主學習引擎，類比實際銀行作業流程，識別潛在風險樣態：

### 4.1 核心演算模型 (Behavioral DNA)
* **時序分析 (RNN/LSTM)**：針對單一帳戶或商戶的交易序列進行建模，識別「快進快出」、「分流後匯集」等典型洗錢路徑。
* **金流拓撲分析 (Graph AI)**：建立跨帳戶的金流關聯圖譜，偵測「接力型轉帳」鏈條（如 A -> B -> C -> D）。
* **聚類行為辨識 (Clustering)**：自動歸納異常交易群聚，對未曾見過的攻擊模式（Zero-day Fraud）進行「常態偏移」警示。

### 4.2 自主成長機制
- **動態基準線 (Dynamic Baseline)**：系統自動學習銀行各時段（平日、假日、連假）的正常交易基頻。
- **特徵偏移偵測**：若發現新型態交易報文或異常清算時間，AI 會自動標記並提取新特徵維度供後續觀察。

## 5. 風險評分與警示矩陣

系統產出之 $R\_score$ 公式為：
$$R\_score = \sum (W_i \times \delta_i) + \Gamma$$
*其中 $W_i$ 為時間動態權重，$\delta_i$ 為特徵偏離值，$\Gamma$ 為名單碰撞因子。*

| 等級 | 評分範圍 | 系統狀態描述 | 銀行端建議動作 |
| :--- | :--- | :--- | :--- |
| **Green** | < 60 | 符合歷史常態與白名單模式 | 正常放行，存檔備查 |
| **Yellow** | 60 - 80 | 行為偏離，觸及低權重風險因子 | 標記交易，加強後續行為追蹤 |
| **Orange** | 80 - 95 | 高度疑似異常，觸及保護名單監控項 | 觸發 MFA 認證或轉由人工照會 |
| **Red** | > 95 | 確認為黑名單碰撞或極端攻擊樣態 | 建議暫緩清算並執行風控決策 |

## 6. 名單管理與隱私保護

* **預載架構**：銀行定期透過加密通道（TLS 1.3）將 `Blacklist`、`Whitelist` 及 `Protection List` 同步至 Risk Box。
* **去識別化比對**：所有帳戶資訊在進入 Risk Box 前即進行 Salting Hash。系統不存儲明文帳號。
* **合規審計**：支援數位簽章驗證，確保所有警示記錄（Alert Logs）具備法律效力且不可竄改。

---

## 7. 技術規格與效能要求

* **交易處理量**：單機支援 ≥ 3,000 TPS。
* **解析時延**：從封包抵達到警示產出總延遲 < 5ms。
* **無感監測**：不佔用生產環境網絡帶寬。
* **硬體建議**：
    * 專用 FPGA 或 SmartNIC 網卡進行封包過濾。
    * 多核心 CPU 並行計算架構（Parallel Processing）。

---

## 8. 技術局限性與待優化空間

> **架構師提示**：為維持「不介入決策」與「高合規」特性，本架構在以下情境需格外注意：
> 1. **全加密流量**：若傳輸層採強加密且無中間代理，需配合銀行端解密密鑰或於 SSL Offloader 後方部署。
> 2. **名單同步頻率**：AI 的有效性取決於銀行提供的「保護名單」更新速度，建議採差量同步（Delta Update）。
> 3. **AI 誤報率控制**：針對節慶等異常大流量情境，需預載銀行業務活動行事曆以降低誤報。
