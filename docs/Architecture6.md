# Risk Box：金融通訊協議層 AI 風險感測節點 (Risk-Sensing Node)

## 版本資訊
- **版本**：v2.1.0 (Enterprise Stable)
- **技術層級**：OSI L2 / L3 協議層
- **監測模式**：Pure Observer (隱身監聽模式)
- **設計哲學**：不介入決策、不修改報文、不觸碰個資、自主學習成長。

---

## 1. 系統定位與核心原則

Risk Box 作為銀行核心網路的「旁路感測器」，在不影響交易效能的前提下，提供即時風險透明度。

* **透明接入**：透過 Port Mirroring (SPAN) 或 TAP 接入，對現有架構零侵入，無需配置 IP 即可運作。
* **非侵入式監測**：系統定位為「Observer」，僅產出風險警示（Alert），不具備阻斷功能，確保銀行業務連續性。
* **個資零碰觸**：僅提取 ISO 標準報文欄位特徵，所有敏感欄位（如帳號）在內存中即時進行 Hash 化，不存儲任何 PII。
* **數據主權歸屬**：黑/白/保護名單由銀行提供並預載，Risk Box 負責執行比對與行為建模。

## 2. 系統接線示意圖 (Network Topology)

Risk Box 採用旁路部署，即使設備故障或斷電，銀行交易鏈路依舊 100% 正常運作：

```text
       [ 銀行核心路由器 / 交換機 ]
                 |
        +--------+--------+ (Port Mirroring / SPAN / TAP)
        |                 |
  [ 銀行核心系統 ]   [ Risk Box ] (Sniffer Mode: 只收不發)
  (生產交易流)       (風險特徵分析)
                          |
                          +------> [ 獨立管理網域 ] (Out-of-band)
                                         |
                                  [ 銀行風控中心後台 ]

## 3. 協議解析引擎 (ISO Mapping)

系統針對金融通訊協議進行深度解析，僅提取非感性交易維度，確保在不接觸個資的前提下完成風險特徵識別：

| 監控維度 | ISO-8583 (ATM/卡交易) | ISO-20022 (匯款/清算) | AI 觀察與異常識別重點 |
| :--- | :--- | :--- | :--- |
| **交易金額** | DE-4 | IntrBkSttlmAmt | 監控是否偏離歷史基頻，或出現規避限額的小額拆分交易 |
| **時間戳記** | DE-7 | CreDtTm | 識別非營業時間、深夜異常高頻或極短間隔的連續交易 |
| **商戶類型** | DE-18 | Purp / Prtry | 監控資金是否流向異常分類之商戶或罕見用途 |
| **收發機構** | DE-32 / DE-33 | InstgAgt / InstdAgt | 偵測異常的跨行清算路徑、發款機構跳轉或非法跳板 |
| **對手帳戶** | DE-102 (Hash) | CdtrAcct (Hash) | 以 Hash 值比對銀行預載之黑名單或受保護名單（如高齡帳戶） |
| **終端識別** | DE-41 | PmtTpInf | 偵測同一設備下的多帳戶異動行為或異常地理跳躍 |

## 4. AI 自主成長與行為建模

Risk Box 內建自主學習引擎，透過模擬實際銀行作業流程，識別潛在風險樣態，並具備隨時間自我優化的能力：

### 4.1 核心行為 DNA 模型
* **時序分析 (RNN/LSTM)**：針對單一帳戶或商戶的交易序列進行建模，識別「快進快出」、「分流後匯集」等典型洗錢與異常路徑。
* **金流拓撲分析 (Graph AI)**：建立跨帳戶的金流關聯圖譜，偵測「接力型轉帳」鏈條（如 A -> B -> C -> D）或漏斗型拓撲結構。
* **聚類行為辨識 (Clustering)**：自動歸納群集型異常交易樣態，對未曾見過的攻擊模式（Zero-day Fraud）進行常態偏移警示。

### 4.2 自主成長機制
* **動態基準線 (Dynamic Baseline)**：系統自動學習銀行在不同時段（如發薪日、節慶、夜間離峰）的正常交易基頻。
* **特徵偏移偵測**：若發現新型態交易報文或異常清算邏輯，AI 會自動標記並提取新維度納入觀察。
* **模型強制校準**：為防止自主學習產生偏誤，系統每 24 小時會與銀行端回傳的「最終確認樣本」進行校準與權重修正。

## 5. 壓力測試模擬與硬體規格 (實際銀行作業情境)

為了應對銀行高度波動的交易環境，系統具備「負載自適應」能力，硬體規格針對以下實際情境進行強化：

### 5.1 實際作業情境模擬
| 作業情境 | 交易負載 (TPS) | AI 處理策略與資源分配 |
| :--- | :--- | :--- |
| **日間均值** | 500 - 1,500 | 全維度 AI 分析，執行深度圖計算與行為 DNA 建模。 |
| **發薪/購物尖峰** | 3,000 - 5,000 | 啟動高併發解析，優先執行黑/白名單碰撞與核心維度過濾。 |
| **夜間離峰** | 50 - 200 | 執行模型自主強化學習、日終特徵壓縮與索引重組。 |
| **瞬時極端高峰** | > 8,000 | 啟動環型緩衝保護以防止丟包，暫緩非關鍵性之行為深度分析。 |

### 5.2 建議硬體規格 (針對高負載優化)
* **處理器 (CPU)**: 雙路 Intel Xeon Platinum 或 AMD EPYC (總計 64 核以上)，以支持大規模多線程並行 AI 推論。
* **記憶體 (RAM)**: 256GB DDR5 ECC，確保數百萬筆保護名單可完全加載於記憶體進行高速查詢。
* **專用網卡 (NIC)**: Mellanox ConnectX-6 以上，支援 100G 吞吐及硬體級時間戳 (Hardware Timestamping)。
* **加速單元 (NPU/GPU)**: NVIDIA T4 或專用 FPGA 卡，用於加速圖形卷積與行為鏈條比對。

## 6. 長期運作穩定性機制 (Anti-Crash / Anti-Overflow)

針對長時間運作可能導致的系統潰敗或溢出，設計如下防禦機制：

* **滑動窗口修剪 (Sliding Window)**: 動態行為特徵僅保留 90 天內之活躍數據，過期特徵自動釋放，徹底防止內存溢出。
* **環型緩衝區 (Circular Buffer)**: 封包接收層採用固定長度記憶體池設計，在尖峰時段即使處理不及也會覆蓋最舊封包，確保系統主進程不癱瘓。
* **無狀態數據流 (Stateless Data Flow)**: 本體不儲存原始封包 (Raw PCAP)，僅存儲輕量化特徵向量，杜絕磁碟 I/O 溢出風險。
* **硬體看門狗 (Watchdog)**: 內建監控進程，偵測到異常時會在毫秒級內自動重啟服務或切換至備機。

## 7. 風險評分矩陣 (R_score)

系統產出之評分僅作為銀行決策系統之參考權重：

| 等級 | 評分範圍 | 系統狀態描述 | 銀行端建議動作 |
| :--- | :--- | :--- | :--- |
| **Green** | < 60 | 符合歷史常態 | 正常放行 |
| **Yellow** | 60 - 80 | 行為偏離常態 | 標記關注，加強觀察 |
| **Orange** | 80 - 95 | 疑似異常，觸及保護名單 | 建議觸發 MFA 或人工照會 |
| **Red** | > 95 | 確認為極端攻擊或黑名單碰撞 | 建議執行即時攔阻 |

## 8. 安全與合規性

* **數據脫敏**: 所有入庫數據皆經過 SHA-256 加密。
* **審計不可否認性**: 所有警示記錄皆帶有數位簽章，確保不可竄改。
* **物理隔離支援**: 支援在無網際網路環境下，透過安全介質手動更新名單。

## 9. 架構師檢視 (Technical Review)

1. **挑戰**: 大規模環境下 Hash 碰撞檢查可能產生延遲。
   * **對策**: 引入 Bloom Filter 作為第一層預篩選，減少記憶體訪問頻率。
2. **挑戰**: AI 自主成長可能因極端案例產生模型偏差。
   * **對策**: 定期導入銀行確認之「真值樣本」進行模型強制修正。
